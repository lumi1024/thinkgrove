# ThinkGrove 知识分层规范

本规范用于解决一个核心问题：**知识不是同一种东西**。

当前框架的 `Domain -> Subdomain -> Question -> Source -> Answer -> Citation/Dispute` 模型已经成立，但它默认所有问题都按同一套 durable-knowledge 标准管理。现实中，同一套问答资产里会混入：

- 稳定可沉淀的方法论
- 带时间窗口的趋势判断
- 依赖个人经验的踩坑总结
- 可直接落地的操作清单
- 社区资源、人脉对接、合作信息

如果不做分层，会出现：
- 时效类内容污染 canonical 质量
- 经验类问题被错误要求严格 citation
- 社区资源被当成知识题治理
- curator 被不同标准的问题压垮

本规范提供最小、可落地、向后兼容的分层方案。

---

## 1. 核心原则

1. **同一棵树，不同资产类型**：ThinkGrove 仍然保持 `question-first` 主结构，但允许同域内混合知识资产。
2. **入口定性质，过程走不同规则**：`Question` 决定这条知识是什么；`Source` 决定它何时有效；`Answer` 决定它需要多强的证据。
3. **最小侵入原则**：不推翻现有 runtime，只在现有 schema 上追加 3 个核心字段。
4. **框架不替产品决定内容**：框架提供分层契约，具体 domain 怎么选类型，由产品侧配置或策展人决定。
5. **可检索、可展示、可治理**：分层不是为了数据库好看，是为了让前端知道怎么展示、让 runtime 知道怎么 curation、让用户知道这条答案可信到什么程度。

---

## 2. 知识资产类型

框架定义 5 种 `content_profile`。它们是问题侧的核心分类，不是 source 分类，也不是 answer 分类。

### 2.1 canonical

- **定义**：durable knowledge，相对稳定、可长期引用、适合沉淀为正式知识
- **例子**：框架契约、MVP 方法、版权基本原则、数据安全最低标准、公司注册/税务常识
- **知识目标**：成为该问题下的长期标准答案
- **时效预期**：中长期有效，靠 freshness decay 管理
- **source 要求**：尽量可验证、可追溯、可复核

### 2.2 situational

- **定义**：时效类知识，只在特定时间窗口内有效
- **例子**：2026 年 AI 创业赛道窗口、某一阶段政策补贴、某个平台算法版本期的运营策略
- **知识目标**：回答“现在/接下来一段时间”该怎么办
- **时效预期**：强依赖时间边界，过期后自动降权
- **source 要求**：优先近期、一手、官方发布

### 2.3 experience

- **定义**：个人/团队踩坑经验、实操总结、非标准化但高价值的方法论
- **例子**：某创始人从 0 到 1 的真实路径、某次失败的复盘、某类合作里的隐性规则
- **知识目标**：降低后来者试错成本
- **时效预期**：部分经验可长期有效，但依赖上下文
- **source 要求**：不强求外部 citation，但必须有清晰 author / 事件背景

### 2.4 operative

- **定义**：可直接落地的操作清单、模板、step-by-step、checklist
- **例子**：AI 内容站 MVP checklist、合规注册步骤、RSS ingestion 配置模板
- **知识目标**：让用户照做就能推进
- **时效预期**：版本化更新；旧版本保留但标注 superseded
- **source 要求**：优先官方文档、一手教程、可复现案例

### 2.5 community

- **定义**：社区资源、对接信息、合作机会、工具/服务/人脉推荐
- **例子**：靠谱的 AI 工具服务商、可对接的渠道资源、社区共建机会
- **知识目标**：降低信息搜寻成本，突出“可用性”和“状态”
- **时效预期**：状态优先，时间次之
- **source 要求**：不强求学术 citation，优先资源状态、对接状态、更新频率

## 3. 三层字段设计

分层不是加一个大字段，而是让 **Question / Source / Answer 三层分别承担不同责任**。

### 3.1 Question：`content_profile`

**作用**：决定这条问题的知识资产类型、治理强度、展示形态和 decay 策略。

**字段**：
- `content_profile`
- 类型：enum
- 可选值：`canonical | situational | experience | operative | community`
- 默认值：`canonical`
- 必填：是
- 可修改：允许策展人在问题生命周期内修改，但必须留 audit trail

**设计理由**：
- 问题决定答案形态
- 也决定 source 准入标准
- 还决定这条内容在首页/森林里的展示策略

### 3.2 Source：`valid_from` + `valid_until`

**作用**：给 source 加时间边界，让时效类知识可以被系统化管理，而不是只靠 `freshness_score` 一个衰减分。

**字段**：
- `valid_from`
  - 类型：datetime，nullable
  - 含义：该 source 开始生效的时间
- `valid_until`
  - 类型：datetime，nullable
  - 含义：该 source 明确失效的时间
  - 如果为 null，表示“暂无明确失效时间”

**设计理由**：
- `freshness_score` 表达“越来越旧”
- `valid_until` 表达“已经失效”
- 两者互补：前者适合 canonical，后者对 situational 是刚需

**规则**：
- `canonical`：`valid_until` 可选；填写时按强规则失效
- `situational`：优先填写 `valid_until`；超过即标记 stale
- `operative`：允许更新版本；原版本可保留历史，但标注 `superseded_by`
- `experience`：不强求，但建议附事件时间
- `community`：资源状态字段优先于时间字段

### 3.3 Answer：`evidence_mode`

**作用**：决定答案的证据要求强度，避免所有答案都用同一套 citation 标准。

**字段**：
- `evidence_mode`
- 类型：enum
- 可选值：`citation_required | experience_allowed | community_vote`
- 默认值：由问题侧 `content_profile` 推导，策展人可覆盖

**映射规则**：
- `canonical` -> `citation_required`
- `situational` -> `citation_required`，但允许“时效说明”作为辅助证据
- `experience` -> `experience_allowed`
- `operative` -> `citation_required` 或 `experience_allowed`，按方法是否可外部验证决定
- `community` -> `community_vote`

**设计理由**：
- 不是所有好答案都必须有 source citation
- 也不是所有经验答案都可以没有作者身份
- `evidence_mode` 让 framework 知道该审什么、展示什么、AI 该怎么生成

## 4. 分类型 curation 规则

### 4.1 通用规则

所有类型共同遵守：
- 不能伪造 source
- 不能伪造作者身份
- 不能把明显过时内容继续当成 canonical 展示
- dispute / vote 机制对所有类型可用

### 4.2 canonical 规则

- source 优先权威、一手、可追溯
- answer 必须显式 citation
- confidence 低于阈值时不允许成为 canonical
- freshness decay 正常生效
- 超过 `valid_until` 若无续订，自动降级或标注 stale

### 4.3 situational 规则

- 问题创建时必须提示填写 `valid_until`
- source 优先近期来源
- answer 必须显式说明时间窗口
- 允许附带不确定性说明
- 超过 `valid_until` 后：
  - 不自动删除
  - 自动降权
  - 展示时突出“已过期”或“仅作历史参考”
- 不允许把 situational 答案长期当作 canonical 展示

### 4.4 experience 规则

- 不强求 source citation
- 必须要求 author 身份清晰
- 必须允许 community dispute
- confidence 与 author reputation / community 反馈共同计算
- 禁止把单一个体经验伪装成统计事实
- 若被 dispute 且裁判成立，必须显式标注“存在争议”

### 4.5 operative 规则

- 优先可执行、可复用、有边界说明
- 允许版本化更新
- 新旧版本之间要显式关联，避免用户引用旧模板
- source 可以是官方文档、一手教程、可复现案例
- 允许 step-by-step / checklist / template 格式
- 更新版本时保留 changelog 或版本说明

### 4.6 community 规则

- 不走 strict citation
- 优先资源状态、对接状态、可用性
- 允许运营/策展人直接更新状态
- 社区资源帖优先 `community_vote`
- 过期资源优先标记“已失效/已下架”，而不是删除
