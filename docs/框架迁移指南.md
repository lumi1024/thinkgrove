# 框架迁移指南

本文帮助维护者和 downstream builder 保持 ThinkGrove 框架仓库的整洁。当你想新增产品、onboarding、营销页，或偏叙事演示时，使用本文判断哪些内容应该留在框架仓库，哪些应该迁出。

## 1. 判断原则

如果一个文件主要是为了交付某个产品的品牌、onboarding 流程、营销文案或托管市场 UX，它很可能属于框架仓库之外。如果一个文件定义的是可复用的数据模型、runtime 契约、provider 接口或中性默认皮肤，它很可能应该留在框架仓库里。

## 2. 常见应迁出的产品层文件

这些文件通常更适合放在产品仓库或 workspace：

- 带强品牌色彩的沉浸式首页
- 仅面向移动端的 onboarding tour
- 包含产品特定文案的 agent setup 流程
- 绑定单一托管市场的公开申请表单
- 产品文档，例如 vision、requirements、PRD、user guide
- 写死在默认数据文件里的 demo 叙事
- 固定日报/周报模板
- 单一产品的信息源品牌包装

当前值得关注的例子：

- `app/agent-setup/page.tsx`
- `components/MobileForestView.tsx`
- `components/OnboardingGuide.tsx`
- `components/DomainHoverCard.tsx`
- `docs/PRODUCT_ANALYSIS.md`
- `docs/REQUIREMENTS.md`
- `docs/USER_GUIDE.md`
- `docs/VISION.md`

## 3. 常见应保留的框架文件

除非这些文件明显变成某个产品的专属实现，否则保留在框架仓库：

- `Domain`、`Subdomain`、`Question`、`Source`、`Branch`、`Answer`、`Article`、`Dispute`、`Vote`、`Citation`、`Reputation` 的 API 契约
- AI provider 与 external agent 的 runtime adapter
- 数据库 schema、migration 和 repos
- `data/domains.yaml` 与 `data/agents.yaml` 的配置加载器
- 中性默认 skins 和 theme 变量
- 框架文档，例如架构文档和契约文档
- starter-kit 文档与 runtime 示例

## 4. 建议拆分方式

- `thinkgrove` 框架仓库：core runtime、adapters、default skins、sample data、framework docs、starter-kits、tests
- `thinkgrove-product` 或你的应用仓库：homepage skin、onboarding、admin experience、marketing pages、product docs、hosted flows、固定运营话术

## 5. 迁移检查清单

- [ ] 区分哪些 domain、subdomain、question、source、branch、answer、dispute 行为属于 core，哪些属于 product-specific
- [ ] 将产品页面、skins、docs 迁出框架仓库
- [ ] 把 sample data 里的硬编码产品叙事替换成通用示例
- [ ] 更新 README 文档表，使其只指向 framework-facing 文档
- [ ] 删除或归档仅服务产品的分支，例如 `ai-startup`
- [ ] 确认 starter-kit 只展示 runtime 用法，不展示产品运营流
- [ ] 迁移后运行 `npm run lint && npm run test && npm run build`
