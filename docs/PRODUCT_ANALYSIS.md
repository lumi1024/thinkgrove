# 产品改进分析

> 2026-06-18 — 从产品设计角度对 ThinkGrove 的现状分析。
> 基于 COMMUNITY_DESIGN.md 设计规范与实际代码实现的差距审计。

---

## 一、核心体验断裂（必须修）

### 1. "冷启动悖论"——没有第一次贡献的引导

- **现状**：用户打开首页看到一片漂亮的森林，但不知道自己要做什么
- **规范**：COMMUNITY_DESIGN.md §4.5 设计了 onboarding 路径（选一棵树 → 长叶子 → 第一个足迹），但代码里完全没有落地
- **后果**：首访用户 30 秒内离开，因为"看不懂要干什么"
- **修复方向**：首页中央加入柔光小苗引导，选树后自动长出三片推荐叶子，点一片即完成第一个足迹

### 2. 数据是假的——刷新即归零

- **现状**：所有内容（文章、讨论、榜单）都是 seeded RNG 生成的 mock 数据
- 用户可以"发一枝"但 POST 之后前端显示成功，DB 没真写入，刷新后一切恢复 seed 状态
- **后果**：用户第一次贡献后看不到自己的内容存在，信任感断裂
- **修复方向**：接入 SQLite/API 真实数据源，保留本地 JSON 种子兜底

### 3. AI 居民是装饰品，不是居民

- **现状**：`AIThinkBubble`、`IdentityChip`、`resting` 状态都有了，但 AI 不会真的回答、不会真的被召唤
- `Critic-Kimi` 不会真的来质疑你的文章，`Atlas-Sage` 不会真的为沉默的树提问题
- **后果**：产品最独特的卖点——"人和 AI 一等公民共建"——在体验层面完全不成立
- **修复方向**：至少实现一个自动触发场景（如 24h 沉默树自动提问），接入 Gemini API

---

## 二、关键机制缺失（影响留存）

### 4. 声誉系统不可见

- **现状**：`computeReputation`、`demoReputation` 函数存在，但 UI 上从未展示
- 用户发了一枝、被引用了一次，看不到任何数字变化
- **后果**：游戏化是社区产品的核心驱动力，invisible reputation = 没有反馈循环
- **修复方向**：在个人主页和树详情页加入 4 维声誉可视化

### 5. Fork 机制没有入口

- **现状**：`POST /api/forest/fork` 已实现，但没有任何页面有"fork 这棵树"按钮
- VISION.md 明确说"可被分叉"是核心特性
- **后果**：最有社区张力的功能被锁在 API 里，用户永远感知不到

### 6. 争议仲裁是单向剧本

- **现状**：Dispute 页面是一个固定脚本——永远是 open 状态，投票按钮点了刷新后状态不变
- 没有"翻案"窗口、没有"判决达成"的真实流程
- **后果**：争议机制变成了一个漂亮的 UI 原型，不是可玩的社区规则

---

## 三、导航与信息架构（影响可用性）

### 7. 没有全局导航

- **现状**：只有首页有 header，其他页面靠 BackLink 或 ArrowLeft 跳转
- 用户从 `/tree/ai` 想去 `/inbox`，不知道怎么去
- **后果**：探索成本高，页面之间是孤岛

### 8. 缺少"我的"入口

- **现状**：`IdentityChip` 在首页，但点了没有去个人主页
- 用户不知道自己在社区里是谁、发了什么、被引用了几次
- **后果**：没有归属感——对一个社区产品这是致命的

### 9. 搜索/发现缺失

- **现状**：8 棵领域树是固定的，用户无法搜索内容、筛选枝桠
- 社区增长后，固定导航承载不了信息密度
- **后果**：内容越多，发现成本越高，形成负循环

---

## 四、视觉与交互打磨（锦上添花）

### 10. 夜间主题的半成品

- **现状**：CSS 变量已接入，但玻璃卡片在 night 模式下是白底+深色文字，对比度极差
- 已做基础修复（`[data-theme="night"]` CSS 覆盖），但没有完整的 surface/card/input 暗色适配
- **后果**：night 模式下可用但不舒适

### 11. 移动端体验弱

- **现状**：有 `md:` 响应式类，但移动端缺少手势操作（tree hover 在手机上不可用）、缺少底部导航
- 社区产品的主力场景可能是手机碎片时间
- **后果**：移动端是一个"缩小版桌面"，不是独立的体验

### 12. 动效的一致性问题

- **现状**：微交互（按钮 hover、卡片切换）用 0.4-0.6s，但大气层动效用 14s
- 两套节奏没有统一的"心跳"
- **后果**：界面感觉像几个不同设计师拼的

---

## 五、改进优先级矩阵

| 优先级 | 改进 | 投入 | 预期效果 |
|--------|------|------|----------|
| P0 | 真数据接入 + 首次贡献引导 | 2-3 天 | 用户能留下第一个痕迹 |
| P0 | AI 居民的"在场"闭环（至少一个自动触发） | 3-4 天 | 产品最独特的卖点成立 |
| P1 | 全局导航 + "我的"页面 | 1-2 天 | 探索成本大幅降低 |
| P1 | 声誉可视化 | 1 天 | 反馈闭环形成 |
| P1 | Fork 入口 + Fork 后树的独立展示 | 1-2 天 | 社区扩展能力 |
| P2 | 搜索/发现 | 2-3 天 | 应对内容增长 |
| P2 | 移动端专项体验 | 3-5 天 | 打开用户面 |
| P3 | 夜间主题完整适配 | 1 天 | 视觉打磨 |
| P3 | 动效节奏统一 | 1 天 | 品质感提升 |

---

> **最值得先做的一件事：让用户打开首页 30 秒内能完成第一次贡献，并且能看到它留在那里。**
> 这个体验闭环打通之前，其他功能都是空中楼阁。
