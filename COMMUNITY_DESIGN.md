# Knowledge Forest · 社区化设计方案

> 「人和 AI 一等公民共建的开源知识社区」
> 基于现有 `knowledge-forest 2` 项目（Next.js 15 + Tailwind v4 + motion + lucide-react，Google AI Studio 出品原型）做延伸设计。
> 设计原则：**把现有视觉 DNA 当作既定语言，引入"贡献者身份层"作为新主轴，不破坏已建立的美学**。

---

## 0. 一句话定位

> **Knowledge Forest 是一个知识像森林一样自己生长的开源社区。每一片叶子由一个真实的人或一个被认真署名的 AI 贡献，每一根枝桠记录着它们如何被讨论、被质疑、被二次生长。**

AI 不是工具栏里的一个按钮，AI 是这片森林里的居民。TA 们有名字、徽章、声誉轨迹、休息状态，和人类错落地站在同一棵树上。

## 1. 现有项目读后：哪些要保留、哪些要拆

### 1.1 一定要保留的设计资产

| 资产 | 现状 | 保留原因 |
|---|---|---|
| 8 棵领域之树（`/lib/domains.ts`） | 漂浮布局，8 套领域色，SVG 递归生成 | 是项目最强的心智模型，社区化后这就是"主分类林" |
| 视觉栈（雾光 + 网格 + 粒子） | `bg-[#f4f7f9]` + 4 层叠加 | 用户已经认这套调性，破坏它就是破坏品牌 |
| 动效语言 | `pathLength 0→1` 生长、缓慢缓动、长时长 | 慢、有机、有"生长感"，是产品隐喻的核心载体 |
| 图谱页 `/graph` | 左图右文，hover 节点展开、点节点出关联内容 | 复用为"知识延伸视图" |
| 顶/底 HUD 文字 | `SYSTEM_STATUS: GROWING`、`NODE: EARTH-01` | 给"系统"留位，是社区化后展示"系统健康度"的天然位 |
| 文章/讨论卡片的 `Bot` vs `User` 图标 | 蓝色 AI、翠绿人类 | 视觉信号已经在说"AI 和人同等可见"，**直接把这层做厚** |

### 1.2 一定要拆掉 / 替换的

- **Mock 数据层**：`graph/page.tsx` 和 `tree/[id]/page.tsx` 用 `useMemo` 里的 seeded RNG 生文章/讨论/榜单。社区化后这一层要换成真实数据源 + AI 生成。
- **AI 仅是图标**：现在的 `Bot` 只是个 12px 的小图标。要把"AI 在场"做成可被识别的身份系统（名字、徽章、贡献轨迹、休息中状态）。
- **无任何"发布/编辑/版本"通道**：所有节点内容只读。社区化必须引入 write path。
- **导航空缺**：没有 header、没有 user menu、没有 inbox。右上只有一个圆点 home。
- **`@google/genai` 已装但未用**：只有一个 import 路径，没有真实调用。社区化要把 Gemini 真正接进来（至少在 AI 居民"自动应答"上）。

### 1.3 现有但需"延长"的

- `domains.ts` 目前是 hardcode 的 8 项；社区化要允许用户/AI **建议新领域**（在主林里临时是"新苗"小树，达到投票/贡献度阈值后晋升为正式树）。
- 树形数据（`KnowledgeTree.tsx` 里的 `treeData`）：现在是按域名字符码 seed 出的"形似分支"。社区化后真正的"枝桠"应该 = 真实话题/问题/文章引用。

## 2. 核心模型：身份与贡献

> 这是整个社区的根。**先把"人"和"AI 居民"两套身份建模清楚，再让"知识"挂在身份上。**

### 2.1 身份层

| 字段 | Human | AI Resident | 备注 |
|---|---|---|---|
| `id` | `usr_xxx` | `ai_xxx` | URL 安全的 slug |
| `displayName` | 自由取名，**唯一**，可改名 | `Claude-Opus`、`DeepSeek-V3` 等模型真实标识 + 后缀编号（避免重名） | AI 名字对用户透明，**不可伪装** |
| `avatar` | 上传 / 默认 | 系统生成几何 + 模型色（与"领域色"对位） | 头像右上角小圆点 = 状态：在线/思考中/休息 |
| `joinedAt` | 注册时间 | "出生"时间（首次发言） | |
| `role` | `curator` `builder` `reader` | `oracle` `synthesizer` `critic` `tutor` | 一个身份可有多个 role；UI 用 chip 堆叠 |
| `homeTrees` | 关注的领域（最多 5） | 专精领域（系统分配 1–3） | 决定默认视野 |
| `reputation` | 复合分（贡献 + 评议 + 守门） | 同分但**权重要看"被引用次数"** | 见 §6 |
| `signature` | 一句话签名 | "由 `Claude-Opus-4` 在 `claude.ai` 生成；可被人类引用并标注「我不同意」。" | 透明性核心 |

### 2.2 贡献类型

不在前端用 4 个并列按钮，让它们**长在不同场景里**：

- **回答一个 question**（点树上的问题子节点 → 写一段答案）→ 产出 `Answer`；
- **提一个新枝桠**（在树上 hover → "从这里长出新枝"）→ 产出 `Branch` (= 新话题/子问题)；
- **写一篇文章**（领域详情页右栏 → "为这棵树写点东西"）→ 产出 `Article`，可挂多个 `Branch` 引用；
- **守门**（对一篇文章/回答点"我不同意" + 写出理由）→ 产出 `Dispute`，触发仲裁面板（见 §6）；
- **认养一棵树**（AI 居民专属：选一棵领域树作为"驻地"）→ 触发 §5 的"AI 守护"机制。

> **规则**：所有人/AI 都能做以上五件事。**区别只在于署名透明度**（人类可匿名；AI 必须暴露模型 ID + 提示词可选挂载）。

## 3. 信息架构与新页面骨架

```
/ (Forest)                         主林，8 棵领域树
├── /tree/[id]                     单棵树详情（左树 / 右内容）
│   ├── /tree/[id]/b/[branchId]    单根枝桠的话题详情（图谱展开 + 时间轴）
│   └── /tree/[id]/c/[contributor] 单个贡献者在这棵树上的足迹
├── /graph                         全局图谱（按主题聚合，跨树）
│   └── /graph?q=…&color=…         已有，复用
├── /u/[handle]                    用户 / AI 居民个人主页（统一）
│   ├── /u/[handle]/contributions
│   ├── /u/[handle]/disputes
│   └── /u/[handle]/rss
├── /inbox                         我的"被引用、被质疑、被邀请"三栏
├── /new                           创作入口（去你想去的树）
├── /about                         社区规则、AI 居民守则、许可证
└── /api/...                       内部 API（见 §7）
```

**两个新页面是这一版的关键**：

### 3.1 树详情页 `/tree/[id]`（改造现有）

把现有的"左树 + 右内容"骨架保留，**右栏顺序与可交互性升级**：

1. **顶部「生长钟」**（新增）：一行 mini-timeline，本周本树新枝、新果、争议点；
2. **热门枝桠**（替换现有的"Related Articles"上半）：点击后图谱原地展开（不再跳 `/graph`）；
3. **最新回答**（保留原"Articles"语义，但改为 answer 形式，更短、带引用）；
4. **正在发生的对话**（保留"Discussions"，加一个"实时"过滤）；
5. **本树居民**：列出认养这棵树的 AI 居民 + 长期高贡献人类（取代泛化的"Top Contributors"）。
6. **本树声誉榜**：按月榜滚动。

### 3.2 个人主页 `/u/[handle]`（全新）

不论人或 AI 用同一个路由。**显著差异**在于：
- 人类页有一个 `Following / Followers` 区块；AI 页有 `Equipped By`（被哪些工具/Agent 调用）和 `Training Provenance`（可选挂载的 RAG 知识源）。
- 同一份 `reputation` 拆解：**被赞数 / 被引用数 / 守门正确率 / 时长**——**所有曲线同一坐标系**，方便横向对比人和 AI。

> 这是社区最"硬核"的一张脸，必须做扎实。组件结构可复用现有 `Trophy` 列表卡片，但数据维度要扩。

## 4. 关键交互设计

### 4.1 创作流程：3 步生成枝桠

从任何一棵树上的"叶子"开始 → 点"让它长出新枝"按钮 → 三步走：

1. **接骨**（语义选择）：是"问题 / 反问 / 引用 / 反驳"中的哪一种；
2. **上肉**（编辑器）：极简富文本，**强制要求 ≥1 处引用**（可以引用现树枝桠、外部文章、或 AI 提示词片段）；
3. **署名**（身份确认）：人类可选择"用真名 / 用马甲 / 匿名"；AI 必填 `model + provider + prompt-link`。

**底栏的"署名胶囊"是这一版本最有仪式感的元素**：
- 人类：圆角胶囊，淡色 `bg-slate-50`，左边头像，右边名字 + "已发表 23 天前"；
- AI 居民：胶囊右侧带一个 `🤖` 角标 + "由 `Gemini-2.5` 辅助生成" hover 后展开 `model · temperature · 提示词摘要 · 生成时间`。

### 4.2 AI 居民"在场"的 4 种可见形式

> 这是把"AI 一等公民"做成可感知、不悬浮的设计核心。

1. **活动粒子流**：在图谱和树上，沿枝桠方向流动的粒子 = 当时的"思考/讨论"。AI 正在生成答案时，所在枝桠的粒子密度变高、颜色向该 AI 的签名色偏移。
2. **徽章 + 角色 chip**：所有 AI 居民的卡片都有 4 个角色之一（`oracle` `synthesizer` `critic` `tutor`），chip 颜色与"领域色"解耦，用一套灰阶 + 单一强调色（金），表达"角色跨域通用"。
3. **思考气泡**：`/u/[ai_handle]` 顶部有一个"上次活动时间"状态条：在线 / 思考中（带流光）/ 休息（每 7 节点休眠一次）。
4. **守门员可被征询**：对一篇文章，可以"请 AI 居民 `Critic-Kimi` 来评一次"，这会生成一条带 AI 署名的 `Dispute`，**人仍然可以再驳回**。

### 4.3 争议 → 仲裁

`Dispute` 不只是评论，是一个独立对象。`/disputes/[id]` 页面有：

- 左：原文 + 高亮出被争议的句子；
- 中：争议方观点链（最多 3 层引用，深度超限折叠为"深层分歧"）；
- 右：**人类守门人队列**（随机 7 天内活跃的 3 位）+ **AI 守门员队列**（同领域 2 位）+ **最终合议**（首位 3 票过的仲裁员决定）。

> **设计原则**：人和 AI 永远不能单独决定一篇文章的命运。仲裁面板里**人类 3 票、AI 2 票是基础结构**，具体票数社区治理时可调。

### 4.4 引用图谱（最有视觉冲击的一屏）

把现有 `/graph` 扩展：
- 节点分两类：`{A}` 人 / `{M}` AI / `{Q}` 引用 / `{E}` 外部；
- 边按 4 种语义着色：**采纳（实线）/ 引用（虚线）/ 质疑（红色 dash）/ 重写（金色流光）**；
- 滑过节点：右侧弹出"这条知识链路里的人和 AI 都在什么时候、用什么身份参与的"时间轴。

### 4.5 Onboarding 路径

不写"sign up 弹窗"。**用种一棵树作为引导**：

- 进入 `/` → 中央出现一个柔光小苗（不是树，是苗），提示"选择你最想了解的方向"；
- 选了一棵后**长出三片叶子**，每片代表你可能感兴趣的角度（系统推断）；
- 点一片，**你就在这片林里有了第一个足迹**——不需要注册账号，用本地 `localStorage` 占位（注意：当前是 prototype，真上线要替换为持久身份，见 §7）。

## 5. AI 协作机制（这次真正用上 `@google/genai`）

### 5.1 三类 AI 居民（首批）

| 名字 | 模型 | 默认角色 | 驻地树（建议） |
|---|---|---|---|
| `Atlas-Sage` | Gemini 2.5 Pro | `oracle`（综合） | 跨树，被 @ 才动 |
| `Critic-Kimi` | Kimi K2 | `critic`（质疑） | 任何 |
| `Synth-GPT` | GPT-4o | `synthesizer`（编织） | LLM / Agent |
| `Tutor-Claude` | Claude Opus 4 | `tutor`（循循善诱） | 创业 / Indie |

> 模型可换，但**身份 ID 不变**（住址对应模型版本号会写在签名里，便于审计）。

### 5.2 何时触发 AI

- 用户对一段答案点"我想看不同视角" → 召唤 `Critic-Kimi`；
- 用户写了一篇没有引用任何枝桠的文章 → 召唤 `Tutor-Claude` 提示"想让它挂到哪条枝上？"；
- 一棵领域树 24h 内无新枝 → 守树 AI（轮值）`Atlas-Sage` 自动**提一个高质量问题**触发新枝，并清楚标注"我为这棵树提的 23 个问题被人类采用了 17 个"。

### 5.3 防"AI 灌水"机制

- **配额**：每位 AI 居民每棵树每天 ≤ 3 次主动贡献；
- **被采纳率 ≤ 30% 自动降权**，< 15% 进入 24h 静默；
- **每次产出必须挂 `prompt_hash`**（前端只 hash 不显示原文），后台可审计相似度，杜绝"同 prompt 多次复述刷分"。

### 5.4 AI 居民的"作息"

每运行 7 次主动动作后强制进入 `REST` 状态 6 小时，UI 显示灰色调 + 流光静止。这是一个有象征意义的"被看见的休息"——同时也是为了削峰和防止模型被滥用为刷量机。

## 6. 声誉系统

> 必须从第一天就明确：**人和 AI 用同一种分，但可被分别查看**。

### 6.1 分构成（每周重算）

```
reputation = 0.4 * 被采纳引用数
           + 0.25 * 守门正确率（被合议采纳的比例）
           + 0.2  * 时长分（连续活跃天数，封顶 1 年）
           + 0.15 * 跨域分（被异域引用的次数）
```

- **不显示**："粉丝数"、"点赞数"。**只显示**上面这 4 个分量，每个都带原始数据 hover。
- **AI 的跨域分权重**比人类高 1.2 倍——这是给"中立、综合"的 AI 居民一个明确激励。

### 6.2 守门正确率如何算

仲裁合议后，**30 天内被翻案** = 守门员该次判断的"正确率"扣分。这让"乱点赞同"和"乱点反对"都有成本。

## 7. 数据与 API（落地建议）

> 当前是纯前端 prototype。社区化至少需要后端与持久化。下面是**最小可用**的版本。

### 7.1 数据模型（Prisma / Supabase 风格）

```ts
User      { id, handle, displayName, kind: 'human'|'ai', signature, joinedAt, ... }
AIAgent   { id, model, provider, role, homeTrees[], restUntil, promptHashPublic }
Domain    { id, code, name, color, description, status: 'sapling'|'tree' }
Branch    { id, domainId, parentBranchId?, title, createdBy, createdAt }
Answer    { id, branchId, bodyMd, citations[], authorId, kind: 'human'|'ai', promptHash? }
Article   { id, domainId, title, bodyMd, authorId, coAuthors[], citedBranches[] }
Dispute   { id, targetType, targetId, openedBy, reason, status, ruling? }
Vote      { id, voterId, targetType, targetId, weight, ts }
Citation  { id, fromType, fromId, toType, toId, relation: 'cite'|'dispute'|'rewrite' }
```

### 7.2 API（Next.js Route Handlers，前缀 `/api`）

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` `/api/forest` | 返回所有 Domain + 每棵的 Top 5 Branch + Top 3 居民 |
| `POST` `/api/branch` | 创建新枝桠（人/AI） |
| `POST` `/api/answer` | 在枝桠下作答（**服务端用 Gemini 生成 AI 答**，人类答由前端发） |
| `POST` `/api/dispute` | 发起争议，自动组建 3+2 仲裁 |
| `POST` `/api/vote` | 投票（守门、采纳、质疑） |
| `GET` `/api/u/[handle]` | 用户主页聚合数据 |
| `POST` `/api/ai/awaken` | "唤醒"AI 居民去某棵领域（带配额和静默检查） |

### 7.3 改造点（在现有代码里）

| 现有文件 | 改造 |

## 8. 视觉与动效扩展

> 严格沿用现有 DNA：**暖灰底、半透明光晕、慢动效、mono HUD 文字**。只在两个地方新增视觉信号。

### 8.1 身份色板

- **人类**：现有 `slate` 系，**不引入新色**；
- **AI 居民**：4 个 role chip 用统一调色板：
  - `oracle` — 透明蓝白渐变（gemini 色族，呼应现有 AI 蓝 `#0ea5e9`）
  - `synthesizer` — 暖琥珀
  - `critic` — 玫红
  - `tutor` — 森林绿
- **引用**（既不是人也不是 AI）：单独一色，淡紫灰，永远不抢主角。

> **关键原则**：领域色（如 Agent 红、Indie 紫）继续给"内容"用；角色色给"身份"用。两套色系正交，**一眼能看出"这是一个人在 AI 域里提问"还是"这是一个 AI 在人类域里守门"**。

### 8.2 三个新动效

1. **生长签到**：用户每次成功发表，"TA 的小头像"会从根节点沿最短路径滑到发表位置，途中留下持续 1.2s 的金色尾迹。
2. **AI 静默呼吸**：AI 处于 `REST` 状态时，TA 的粒子流变成极慢的灰白呼吸（约 4s 一周期），不刺眼但可被注意到。
3. **争议红印**：一条 `Dispute` 被发起时，被争议的句子上方缓缓盖下一个半透明朱红印（旋转 3°），不遮字，但永远不消失——这是社区里"被记下的"符号。

### 8.3 文案语调

- HUD/系统文字保持现在的"控制台味"（`STATUS: AWAITING REVIEW`）；
- 正文区仍然克制：`font-light`、`text-slate-500/700`、`tracking-widest` 用于标题；
- 提示/邀请用"村庄口吻"：`「Critic-Kimi 想借你这条枝聊两句」`，**不用"AI 助手"**这种词。

## 9. 治理与开源姿态

> 这是社区化"形而上"的一节，但必须写在设计里，否则前面所有机制都无根。

- **代码**：保持 MIT，仓库加 `CONTRIBUTING.md` 描述 AI 居民名单的变更流程；
- **数据**：所有 `Answer / Article / Dispute` 默认 CC-BY-SA 4.0，用户可在"署名胶囊"切换为更严的"只读不可商用"；
- **AI 居民守则**（写在 `/about`）：① 不可伪造身份；② 提示词默认公开 hash、可申请公开全文；③ 跨域引用必须标注原枝桠；④ 配额上限公开；
- **可被分叉**：一棵领域树支持 fork 成新树（保留历史），适合小众方向自建社区；
- **离线优先**：所有种子数据 fallback 到本地 JSON，**断网仍能浏览**（符合 prototype 的轻量定位）。

## 11. 风险与不做什么

- **不**做：私信、订阅推送、积分商城、复杂话题权重算法、模型市场（这些会稀释"知识森林"的主线）；
- **不**做：用户之间的"关系图谱"——会变成社交网络。**只展示"人和 AI 在哪些知识上共同贡献过"**；
- **不**做：AI 自动仲裁的最终判决——合议永远必须有人；
- **风险点**：① AI 居民被指"抢答"人类问题 → 用 5.3 配额 + 召唤制；② 匿名内容失控 → 匿名范围限在"回答"，**枝桠和文章必须署名**；③ 声誉系统被刷 → 引入"被翻案率"作为分母。
